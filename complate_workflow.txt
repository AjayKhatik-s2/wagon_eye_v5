  PURPOSE
  -------
  Every component in this pipeline is built around ONE invariant:

      GlobalTrainState is the single source of truth for wagon counting,
      numbering, boundaries, and ENGINE / WAGON / BRAKE_VAN classification.

  Only Stage 1 (the wagon_count subpackage) is allowed to detect gaps,
  perform counting, assign wagon ids, or define wagon boundaries.  Every
  later stage consumes the GlobalTrainState and produces a deterministic
  artefact derived from it -- never a recount.

  RIGHT_UP is the master camera.  Its gap detections and classifications
  drive the GlobalTrainState; the other three cameras contribute support
  evidence only.

  Each feature processor reuses the SAME mature inference intelligence
  the legacy system spent thousands of lines on -- Kalman + Hungarian
  tracking, illumination quality scoring, geometric shape priors,
  state-machine hysteresis with quality-weighted voting, identity
  merging, OCR preprocessing + digit-level voting, edge-zone damage
  suppression, loaded-wagon floor-damage filtering, etc.  These live
  in `features/inference_lib/` and are driven from a pure frame
  iterator -- never from a `cv2.VideoCapture` loop.

================================================================================
  ARCHITECTURE INVARIANTS
================================================================================

  +----------------------------------------------------------------------+
  | 1. GlobalTrainState is IMMUTABLE downstream of Stage 1.              |
  | 2. Frames are extracted EXACTLY ONCE FOR INFERENCE (Stage 2).  No    |
  |    detector / YOLO / OCR / tracking call exists outside Stage 1      |
  |    (subprocess) and Stage 3.  Stage 4b's overlay renderer DOES open  |
  |    the raw videos with cv2.VideoCapture, but strictly for            |
  |    visualization: it never invokes any model -- every overlay is     |
  |    drawn from already-persisted GlobalTrainState, UnifiedWagonState, |
  |    evidence metadata, and per_camera_tracking.json.                  |
  | 3. Each feature processor is driven from a pure frame iterator over  |
  |    the cached JPEGs.  No process_video() wrappers.                   |
  | 4. The frame loop hands every detection through the SAME mature      |
  |    legacy intelligence (Kalman tracking, identity merging, quality   |
  |    scoring, FSM voting, etc.) that the old camera-centric code used. |
  |    No simplification, no shortcuts -- the inference quality matches  |
  |    the legacy system.                                                |
  | 5. Models live in ONE centralized tree.  No per-camera duplication.  |
  | 6. Legacy camera folders (RIGHT_UP/, LEFT_UP/, RIGHT_UP_TOP/,        |
  |    LEFT_UP_TOP/) and the legacy process_video / DoorProcessor /     |
  |    DamageProcessor wrappers do NOT exist inside this package.        |
  | 7. Stage failures are LOCAL:  per-wagon failure -> NO_DATA cell.     |
  |    Only Stage 1 failure aborts the batch.  Stage 4b (rendering) and  |
  |    Stage 5a (camera reports) failures are per-artifact: a            |
  |    failed camera mp4 or camera PDF does not block the rest.          |
  +----------------------------------------------------------------------+

================================================================================
  STAGE 0  (orchestrator) -- batch acquisition
================================================================================

  orchestrator/master_runner.py acquires one TrainBatch.  Two modes:

  AUTO    Poll all 4 S3 input prefixes, cluster videos by filename
          timestamp into batches, pick the oldest complete (or aged
          partial) batch (delegates to train_batch_manager).

  LOCAL   Scan a local folder for one .mp4 per camera (filenames must
          contain "right_up" / "left_up" / "right_up_top" / "left_up_top").

  All 4 videos downloaded (or pass-through for local) to
  <workspace>/<batch_key>/downloads/.

  If fewer than 4 cameras are present, Stage 1 is skipped and the
  batch is marked failed_no_global_state immediately.


================================================================================
  STAGE 1  reconstruction/runner.py -- global train reconstruction
================================================================================

  Subprocess wrapper around wagon_count/run_global_count.py with flags:
      --no-frames   (materializer owns frame extraction)

  Note: --no-videos is intentionally NOT passed since v4.2 so that
  wagon_count emits its per-camera tracking overlay mp4s as
  debug artifacts (gap boxes + wagon-boundary flashes only, no feature
  annotations).  These are SEPARATE from the rich Stage-4b overlays:

      global_state/processed_videos/<CAM>_processed.mp4
          wagon_count debug overlays (Stage 1 artifact)
      processed_videos/<CAM>_processed.mp4
          rich feature overlays (Stage 4b artifact -- this is what the
          combined report's "PROCESSED VIDEO" URLs point to)

  wagon_count internally runs:
      per-camera gap tracking (Kalman + persistence)
      RIGHT_UP master ENGINE / WAGON / BRAKE_VAN classification
      cross-camera gap fusion (recover gaps RIGHT_UP missed if >=2
                               support cameras agree)
      GlobalTrainState assembly with deterministic GW_n ids

  Model resolution accepts both the legacy long names
  (right_up_wagon_gap.pt / left_up_wagon_gap.pt) AND the new short
  names (right_up_gap.pt / left_up_gap.pt) -- transparent fallback
  inside wagon_count._resolve_model().

  Output consumed:
      global_state/global_train_state.json    -> GlobalTrainState
      global_state/per_camera_tracking.json   -> {camera_id -> fps,
                                                  gaps, dimensions}

  Failure policy:
      subprocess exit != 0     -> ReconstructionError -> batch aborted
      JSON file missing        -> ReconstructionError -> batch aborted
      total_wagons == 0        -> ReconstructionError -> batch aborted
      Batch final_status: failed_no_global_state.  No PDF, no email.


================================================================================
  STAGE 2  materializer/wagon_cache_builder.py
================================================================================

  Single-pass per-video frame extraction into the shared cache.

  For each camera (parallel via ThreadPoolExecutor(4)):
      1. Open the source video ONCE via cv2.VideoCapture.
      2. Pre-compute each GW_n's camera-local frame window:
                local_start = round(gw.start_time * local_fps)
                local_end   = round(gw.end_time   * local_fps) - 1
         clipped into [0, total_frames - 1].
      3. Walk frames sequentially; write each frame's JPEG to
                wagon_cache/<GW_n>/<camera_folder>/frame_<NNNNNN>.jpg
         (camera_folder is lowercase: right_up, left_up, right_up_top,
         left_up_top).

  This is the LAST place in the pipeline that decodes video.  Every
  later stage reads JPEGs from disk only.

  Failure policy:
      One camera's video missing / unreadable -> empty subtree for that
                                                  camera; other cameras
                                                  still populated.  Batch
                                                  continues as
                                                  completed_partial.
      Materializer throws (unexpected)        -> failed.  Abandoned.

================================================================================
  STAGE 3  feature inference -- features/{door,ocr,load,damage}/processor.py
================================================================================
================================================================================

  Parallel execution across the 4 features via
  ThreadPoolExecutor(max_workers=4).  Wagons within a feature are
  iterated SERIALLY so each YOLO model and each easyocr Reader is
  loaded ONCE per process and reused across every wagon.

  Each processor is a thin driver that wraps the cached-frame loop
  around the MATURE LEGACY INTELLIGENCE living in
  features/inference_lib/.  No detector code is re-implemented; the
  legacy classes are simply called from a frame iterator instead of a
  cv2.VideoCapture loop.


================================================================================
  3.A  DOOR  features/door/processor.py
================================================================================

  Per wagon, for each side camera (RIGHT_UP, LEFT_UP), runs the full
  legacy door-intelligence chain:

  +----------+ frame +----------------+ +-------------+ +------------+
  |          | ----> | IlluminationPr | | YOLO        | | Geometric  |
  | iter     |       |  ocessor       | | door_state  | | ShapePrior |
  | cached   |  +--> | .process_frame |->| .pt (det)  |->| .filter    |
  | frames   |  |    | quality_score  | | half=True   | | _detections|
  +----------+  |    +----------------+ +-------------+ +------------+
                |                                              |
                |                                              v
                |                                       +------------+
                |                                       | DoorTracker|
                |                                       | .update    |
                |                                       | (Kalman +  |
                |                                       | Hungarian +|
                |                                       | TrackMemory|
                |                                       | + FSM)     |
                +<--------------------------------------+------------+
                                                               |
                                                  after-loop   v
                                                       +----------------+
                                                       | get_final_door |
                                                       | _states +      |
                                                       | IdentityMerger |
                                                       +----------------+
                                                               |
                                                               v
                                                  _pick_side_state()
                                                  (DAMAGED > OPEN >
                                                   PARTIAL > CLOSED)

  Detailed per-frame work:

      1) IlluminationProcessor.process_frame(frame)
         Quality score in [0, 1] from:
            brightness    weight 0.20  (optimal ~128, penalize >=200)
            glare         weight 0.35  (saturated >=250 pixels)
            shadow        weight 0.35  (dark <=30 pixels)
            uniformity    weight 0.10  (texture std-dev)
         Note: legacy explicitly skips photometric correction BEFORE
               YOLO inference (`door_processor.py:1076-1080`).  Quality
               score is used ONLY to weight the tracker's voting.

      2) YOLO door_state.pt(frame, verbose=False, half=True)
         Returns boxes + class names from:
            open / open_door     -> OPEN
            closed / closed_door -> CLOSED
            partial_closed / closed_with_wire -> PARTIAL
            damage               -> DAMAGE
         Confidence floor applied EARLY in the processor:
            conf >= closed_confidence_threshold (default 0.69)
         Higher per-class thresholds enforced INSIDE the tracker:
            OPEN requires conf >= open_confidence_threshold (default 0.77)

      3) GeometricShapePrior.filter_detections(frame, boxes, confs, cls)
         Three checks (all relaxed defaults; non-blocking gates):
            * aspect ratio in [0.15, 2.0], preferred ~0.50
            * vertical-edge dominance Sobel_x / Sobel_y >= 0.80
            * border-completeness via Canny >= 0.01 edge density

      4) yolo_to_detections(...) wraps survivors in Detection objects
         and tags each with `illumination_quality` so the tracker can
         quality-weight votes.

      5) DoorTracker.update(detections, frame, frame_w, frame_h):
            a) Kalman predict for all active tracks (constant-velocity,
               adaptive process noise: stopped=0.01 / moving=0.5)
            b) Hungarian association with multi-term cost:
                  cost = 0.5 * normalized_distance
                       + 0.5 * (1 - IoU)
                       + 0.2 * (1 - HSV-histogram-similarity)
               max_center_distance = 150 px (gate)
               min_iou_threshold   = 0.10  (gate)
               direction gate rejects backward matches (>10 px wrong way)
            c) For matched tracks:
                  - Kalman update
                  - TrackMemory.add_prediction with quality-weighted entry
                  - StateMachine.update with streak counters
                  - Snapshot scoring (defer; see step 7)
            d) For unmatched tracks: mark_lost; check exit gate.
            e) For unmatched detections: try revive a deleted track
               (spatial overlap + context similarity); else open a new
               tentative track.
               n_init = 2 hits to confirm; max_age = 30 frames.

      6) Per-track StateMachine in door_tracker.py (lines 338-510):
            UNKNOWN -> OPEN              needs open_streak >= 5
            UNKNOWN -> PARTIAL_CLOSED    needs partial_closed_streak >= 4
            UNKNOWN -> CLOSED            needs closed_streak >= 4
            CLOSED  -> OPEN              needs open_streak >= 5
            OPEN    -> CLOSED            needs closed_streak >= 8 *
            OPEN    -> PARTIAL_CLOSED    needs partial_closed_streak >= 8 *
            DAMAGE  is TERMINAL          (resets all other streaks)
              (*) 2x hysteresis -- legacy bias against false-closes
                  on safety-critical OPEN doors.

      7) Snapshot scoring (deferred until end of loop):
            score = 2.0 * bbox_area_ratio
                  + 2.5 * (1 - horizontal_distance_from_center)
                  + 1.0 * detection_confidence
                  + 0.5 * quality_score
                  * 0.30 if bbox center within 8% of frame edge

      8) Per-track final decision (door_tracker.py:835-909):
            * Get majority class + weighted confidence from TrackMemory.
            * If hits < min_hits_for_decision (2), use majority directly.
            * Otherwise feed majority + weighted confidence into FSM.
            * If FSM says OPEN but weighted_conf < 0.77, DOWNGRADE to
              PARTIAL_CLOSED for safety-conservative reporting.
              (door_tracker.py:886-887)

  After the frame loop:

      9) DoorIdentityMerger.merge_all_tracks(all_tracks_objs)
         Post-tracking pass that collapses fragmented tracks of the
         SAME physical door using four similarity measures (weights
         from MergeConfig):
            spatial    0.40  (normalized center distance; hard gate 0.50)
            temporal   0.15  (overlap + adjacency within 30 frames)
            context    0.25  (HSV histogram correlation)
            structural 0.20  (128x128 grayscale cross-correlation)
         Merge if combined >= 0.85 (conservative).

  Per-camera/side picker (`_pick_side_state` in the processor):

      Priority:  DAMAGED  >  OPEN  >  PARTIAL  >  CLOSED  >  NO_DATA
      Tiebreak inside a class:  (total_hits, confidence).

      RIGHT_UP -> right_door
      LEFT_UP  -> left_door

  Output JSON shape (one per wagon):

      {
          "global_id": "GW_7",
          "feature":   "door",
          "status":    "OK" | "NO_FRAMES" | "FAILED" | "NO_DATA",
          "left_door":              "CLOSED | OPEN | PARTIAL | DAMAGED",
          "left_door_confidence":   0.91,
          "right_door":             "...",
          "right_door_confidence":  0.83,
          "tracks": [
              {camera_id, track_id, state, confidence, first_frame,
               last_frame, total_hits, mean_center_x},
              ...
          ],
          "supporting_cameras": ["LEFT_UP", "RIGHT_UP"],
          "frame_count": ...,
      }

  Failure policy:
      door_state.pt missing             -> NO_DATA for every wagon.
      Per-wagon exception (e.g. corrupt JPEG inside one wagon folder)
                                        -> {"status": "FAILED", ...} for
                                           that wagon; rest continues.


================================================================================
  3.B  OCR  features/ocr/processor.py
================================================================================

  Per wagon, runs the full legacy OCR intelligence chain on RIGHT_UP
  frames ONLY (RIGHT_UP is the master / OCR authority).  ENGINE and
  BRAKE_VAN wagons are short-circuited (no payload OCR).

  +----------+ frame +-------------+ +-----------------+ +-------------+
  |          | ----> | YOLO        | | WagonNumberOCR  | | WagonNumber |
  | iter     |       | wagon_id_   | | .reconstruct_   | | Aggregator  |
  | cached   |       | counting.pt | | wagon_number    |->| .add_wagon  |
  | RIGHT_UP |       | det boxes   |->| (full 6-step    | | _number     |
  | frames   |       +-------------+ |  preprocessing  | +-------------+
  +----------+                       |  + easyocr +    |        |
                                     |  validation +   |        v
                                     |  reconstruction)|+--------------+
                                     +-----------------+| get_aggregat |
                                                        | ed_numbers   |
                                                        +--------------+
                                                                |
                                                                v
                                                   pick (is_full_length,
                                                          observations,
                                                          mean_conf)

  Per-frame YOLO + per-crop OCR pipeline:

      1) YOLO wagon_id_counting.pt(frame, verbose=False, half=True)
         Every detected bbox above det_confidence (default 0.40) is a
         candidate wagon-number region.  No NMS or spatial dedup at
         frame level -- all crops are independently OCR'd.

      2) For each surviving bbox:
            crop = features._common.crop_bbox(frame, bbox, pad=10)
            wagon_num = WagonNumberOCR.reconstruct_wagon_number(
                crop, yolo_conf, debug=False
            )

      3) Inside reconstruct_wagon_number, the legacy 6-step ROI crop
         preprocessing runs first:
            i)   pad with 10 pixels (already done in step 2)
            ii)  3x cubic upscale  (cv2.INTER_CUBIC)
            iii) NLMeans denoise  (h=8, hColor=8,
                                   template=7x7, search=21x21)
            iv)  CLAHE on the LAB L-channel (clipLimit=3.5,
                                              tileGridSize=8x8)
            v)   unsharp masking  (1.5x weight, sigma=3 blur)
            vi)  RGB conversion   (easyocr expects RGB)

      4) easyocr.Reader.readtext(image,
                                 allowlist='0123456789',
                                 paragraph=False,
                                 detail=1)
         Digit-only mode; per-character confidence returned.

      5) Multi-line reconstruction (wagon_number_ocr.py:275-283):
         Sort recognized texts top-to-bottom by Y coordinate and
         concatenate digits; pick the rightmost 11-digit window if the
         concatenated string is longer.

      6) Wagon-type confusion-map correction
         (wagon_number_ocr.py:319-368):
         If the first 2 digits fall outside the valid Indian Railways
         wagon-type range [10, 39], apply the OCR confusion map:
            '0' -> '2'   '4' -> '1'   '5' -> '3'
            '6' -> '1'   '7' -> '1'   '8' -> '3'   '9' -> '3'
         Returns a `was_manipulated` flag for audit.

      7) WagonNumberValidator.validate(digits, ...):
            * len(digits) == 11                         hard
            * digits.isdigit()                          hard
            * first two digits in [10, 39]              hard
            * year code (digits[4:6]) in 1950-2049      warn-only

      8) Build a WagonNumber dataclass with:
            full_number, wagon_type, owning_railway, year_of_manufacture,
            individual_number, check_digit, yolo_confidence, ocr_confidence

      9) WagonNumberAggregator.add_wagon_number(wagon_num, frame_idx):
         Stores per-string buckets with running statistics.

  After the frame loop:

      10) WagonNumberAggregator.get_aggregated_numbers() returns
          AggregatedWagonNumber records, each carrying:
             * exact-string grouping
             * digit-level voting per position when multiple results
               disagree by 1-2 digits (Counter.most_common per index)
             * frame_count >= min_frame_count (2)
             * avg_confidence >= min_confidence (0.30)
             * `require_validation = True` keeps only validated entries

      11) Processor picks the winner with sort key
              (is_full_length, observations, mean_conf, full_number)
          and reports it as the wagon_identifier.  Aggregator output
          plus the first 8 raw frame candidates are kept in the JSON
          for audit.

  Output JSON shape (one per wagon):

      {
          "global_id":  "GW_7",
          "feature":    "ocr",
          "status":     "OK" | "NO_FRAMES" | "FAILED" | "NO_DATA",
          "wagon_identifier":              "32145678901",
          "wagon_identifier_confidence":   0.83,
          "candidates":  [
              {full_number, observations, mean_conf, yolo_conf,
               is_full_length},
              ...
          ],
          "raw_candidates_first_8": [...],
          "supporting_cameras": ["RIGHT_UP"],
          "frame_count": ...,
      }

  Failure policy:
      wagon_id_counting.pt missing OR easyocr unavailable
                                  -> NO_DATA for every wagon.
      ENGINE / BRAKE_VAN          -> OK with wagon_identifier=NO_DATA
                                      and skipped_reason=classification.
      Per-wagon exception         -> FAILED for that wagon.


================================================================================
  3.C  LOAD  features/load/processor.py
================================================================================

  Per wagon, runs the legacy loaded/empty voting on top cameras
  (RIGHT_UP_TOP authoritative, LEFT_UP_TOP supporting).  ENGINE and
  BRAKE_VAN wagons are short-circuited (no payload to classify).

  Per-frame work:

      1) YOLO loaded.pt(frame).  This model emits CLASSIFICATION
         output (`results.probs.top1` + `top1conf`).  features._common.
         run_classification handles the box-fallback gracefully if a
         detection-style variant is loaded instead.

      2) Canonicalize the top-1 class name:
            loaded / load / full   -> LOAD_LOADED
            empty / unload         -> LOAD_EMPTY
            other                  -> not counted

      3) For each camera, accumulate per-class confidence lists:
            loaded_confs : List[float]
            empty_confs  : List[float]
         (Frames that don't map into either bucket are counted in `used`
          but contribute no vote -- mirrors legacy behaviour.)

  After the frame loop, the LEGACY VOTING RULE
  (RIGHT_UP_TOP/damage_processor.py:1047) is applied per camera:

      loaded_ratio = n_loaded / n_frames_used
      if loaded_ratio > 0.35 and n_loaded > 0:
          camera_status = LOADED ; conf = mean(loaded_confs)
      elif n_empty > 0:
          camera_status = EMPTY  ; conf = mean(empty_confs)
      else:
          camera_status = NO_DATA

  Cross-camera fusion (matches the Stage-4 authority rule):

      if right_up_top.status != NO_DATA:
          fused = right_up_top
      elif left_up_top.status != NO_DATA:
          fused = left_up_top
      else:
          fused = NO_DATA

  Output JSON shape (one per wagon):

      {
          "global_id":   "GW_7",
          "feature":     "load",
          "status":      "OK" | "NO_FRAMES" | "FAILED" | "NO_DATA",
          "load_status": "LOADED" | "EMPTY" | "NO_DATA",
          "load_confidence": 0.85,
          "per_camera": {
              "RIGHT_UP_TOP": {load_status, confidence, loaded_count,
                               empty_count, frames_used, loaded_ratio},
              "LEFT_UP_TOP":  {...},
          },
          "supporting_cameras": [...],
          "frame_count": ...,
      }

  Failure policy:
      loaded.pt missing       -> NO_DATA for every wagon.
      ENGINE / BRAKE_VAN      -> OK with load_status=NO_DATA and
                                  skipped_reason=classification.
      Per-wagon exception     -> FAILED for that wagon.


================================================================================
  3.D  DAMAGE  features/damage/processor.py
================================================================================

  Per wagon, runs the full legacy damage intelligence on top cameras
  (RIGHT_UP_TOP + LEFT_UP_TOP).  ENGINE and BRAKE_VAN wagons are
  short-circuited.

  +----------+ frame +-------------+ +------------+ +------------+
  |          | ----> | YOLO        | | filter_dets| | Damage     |
  | iter     |       | damage.pt   | | (conf >=   | | Tracker    |
  | cached   |       | det boxes   |->| 0.55       |->| .update    |
  | TOP      |       +-------------+ |  area 0.5- | | (Kalman +  |
  | frames   |                       |   40%      | | Hungarian +|
  +----------+                       |  skip-class| | voting +   |
                                     |   list +   | | snapshot)  |
                                     |  edge-zone)| +------------+
                                     +------------+        |
                                                           v
                                            +----------------------+
                                            | get_final_damage     |
                                            | _states +            |
                                            | cross-track dedup +  |
                                            | loaded-wagon floor   |
                                            | filter +             |
                                            | any-camera-wins      |
                                            +----------------------+

  Per-frame filtering (_filter_detections_for_top in processor.py):

      For each YOLO detection:
          1. SKIP if class_name in {'no_damage', 'wagon', 'engine',
                                    'tail', 'brake_van', 'guard_van',
                                    'background', 'outer_wall_damage'}
             (outer_wall is a side-camera concern; ignored on TOP.)
          2. SKIP if conf < 0.55
          3. SKIP if bbox_area / frame_area outside [0.005, 0.40]
                (tiny-noise floor, huge-FP ceiling)
          4. EDGE-ZONE suppression with bypass:
                cx_ratio = cx / frame_w
                cy_ratio = cy / frame_h
                at_edge =
                  (cx_ratio < 0.12 OR cx_ratio > 0.88)
                  OR
                  (cy_ratio < 0.10 OR cy_ratio > 0.85)
                IF at_edge AND conf < 0.70:  SKIP   (likely coupling
                                                     / transition zone
                                                     artefact)

  Survivors are wrapped via `yolo_to_damage_detections(...)` and fed
  into the legacy DamageTracker:

      5) DamageTracker.update(detections, frame, frame_w, frame_h):
            * Kalman predict (6D state, adaptive Q like the door tracker)
            * Hungarian association with PURE distance cost:
                  dist_cost = min(dist / 200.0, 1.0)
                  iou_weight    = 0.0   (disabled for moving train)
                  distance_weight = 1.0
                  hard gate at dist > 200 px (cost = 1e5)
            * For matched tracks: predict + update + TrackMemory
              (confidence-weighted majority vote).
            * Best-snapshot selection per track favours non-gap frames
              (`is_gap_frame` flag); backup snapshot stored as fallback.
            * Track lifecycle: n_init=2, max_age=30,
              min_hits_for_decision=2.

  After the frame loop, per camera:

      6) DamageTracker.get_final_damage_states() returns
         {track_id -> {class_name, confidence, best_confidence,
                       total_hits, first_frame, last_frame,
                       best_snapshot, best_snapshot_bbox,
                       best_frame_idx}}.

         If best_snapshot came from a gap frame, the tracker auto-swaps
         it with the backup snapshot.

      7) Cross-track dedup within ONE camera (_dedup_cross_tracks):
            Sort tracks by (-conf, -hits).
            For each track, drop if there is already a survivor with:
                same class_name AND
                (center distance < 100 px  OR  bbox IoU >= 0.20)
            Survivor wins.

  Cross-camera + cross-feature fusion (processor.run):

      8) any_damage = True if either RIGHT_UP_TOP or LEFT_UP_TOP
                       reports >=1 confirmed track.

      9) LOADED-WAGON FLOOR-DAMAGE FILTER (legacy heuristic, exact):
            If our companion `wagon_states/load/<GW_n>.json` exists and
            reports load_status == LOADED, drop every track whose
            class_name == "floor_damage" (the floor is hidden under
            the cargo; such detections are reflections / cargo edges).
            Re-evaluate any_damage after the filter.

     10) Per-wagon top_damage verdict:
            DAMAGE  if any_damage after the loaded-wagon filter
            OK      if frames seen but no surviving track
            NO_DATA if no frames at all

  Output JSON shape (one per wagon):

      {
          "global_id":   "GW_7",
          "feature":     "damage",
          "status":      "OK" | "NO_FRAMES" | "FAILED" | "NO_DATA",
          "top_damage":  "DAMAGE" | "OK" | "NO_DATA",
          "top_damage_details": [
              {camera_id, track_id, class_name, confidence,
               best_confidence, total_hits, first_frame, last_frame,
               best_frame_idx, bbox},
              ...
          ],
          "per_camera":  {
              RIGHT_UP_TOP: {damage_status, frames_used, track_count,
                              tracks},
              LEFT_UP_TOP:  {...},
          },
          "supporting_cameras": [...],
          "frame_count": ...,
      }

  Failure policy:
      damage.pt missing       -> NO_DATA for every wagon.
      ENGINE / BRAKE_VAN      -> OK with top_damage=NO_DATA + skipped_reason.
      Per-wagon exception     -> FAILED for that wagon.

  Execution ordering note:
      The DAMAGE processor reads sibling LOAD JSONs to drive the
      loaded-wagon floor-damage filter.  In the parallel Stage-3
      ThreadPoolExecutor, this means the damage processor MAY observe
      an in-progress wagon_states/load/ directory.  The processor
      `os.path.exists` checks at read time -- if the load file isn't
      there yet, the floor-damage filter is silently skipped.  In
      practice the load feature finishes much faster than damage
      (classification vs detection), so the filter applies to most
      wagons.

  STAGE 4  fusion/wagon_state_builder.py
================================================================================

  Merge the four per-feature JSONs into one UnifiedWagonState per
  GW_n.  Authority rules (codified in build() per spec):

      +------------------------+--------------------------------------+
      | Field                  | Authority                            |
      +------------------------+--------------------------------------+
      | classification         | GlobalTrainState (RIGHT_UP)          |

  Failure policy:

                                    but the batch does NOT abort.

================================================================================
  STAGE 5a  reporting/camera_reports.py  (CAMERA-WISE PDFs)
================================================================================

  Reproduces the legacy WagonEye camera-wise reporting hierarchy.  Four
  reportlab PDFs emitted to reports/ -- ONE PER CAMERA -- built BEFORE
  the combined report so it can link them:

      right_up_report.pdf       authority: right door, OCR, classification
      left_up_report.pdf        authority: left door
      right_up_top_report.pdf   authority: load, top damage
      left_up_top_report.pdf    authority: top-damage support / validation

  Each camera report shows ONLY what that camera is authoritative for,
  and uses snapshots from THAT camera only.

  Page order (A4 landscape):

      1. Camera Summary page
           navy "<CAMERA> CAMERA REPORT" banner + IST timestamp
           "Authoritative for: ..." teal line
           4 KPI cards: VISIBLE WAGONS (n/total), DETECTED ANOMALIES,
                        PROCESSING CONFIDENCE (avg of the camera's
                        authoritative confidences), COVERAGE %
                        (wagons with >=1 cache frame for this camera)
           Detection Summary table (authority-specific metrics)

      2. Per-wagon pages  (anomalous wagons first, then serial order)
           - non-wagon (ENGINE / BRAKEVAN) on top cameras -> single
             simple page ("NOT COUNTED AS WAGON")
           - else: 2x2 quartile wagon-overview page from this camera's
             wagon_cache frames (or "NOT VISIBLE TO THIS CAMERA"), then
             one state-tinted detail page per camera-authoritative
             detection that has a snapshot or an anomaly.

      3. Camera Anomaly Summary
           grouped by severity (HIGH = open door / damage, MEDIUM =
           partial closed, LOW = OCR missing / load no-data), each a
           SR.NO | GW ID | ANOMALY table.

      4. Camera Evidence section
           legacy snapshot grid (info row + 2-column image grid) for the
           camera's flagged wagons.

  Camera authority -> per-wagon content:
      RIGHT_UP      right door (state+conf+snapshot), OCR
                    (number+conf+snapshot), classification confidence.
                    Anomalies: RIGHT_DOOR OPEN/DAMAGE (HIGH),
                    OCR_MISSING on WAGON-class (LOW).
      LEFT_UP       left door.  Anomalies: LEFT_DOOR OPEN/DAMAGE (HIGH),
                    PARTIAL CLOSED (MEDIUM).
      RIGHT_UP_TOP  load status (authoritative) + top-damage tracks whose
                    camera_id == RIGHT_UP_TOP.  Anomalies: TOP_DAMAGE
                    (HIGH), LOAD_NO_DATA on WAGON-class (LOW).
      LEFT_UP_TOP   top-damage tracks whose camera_id == LEFT_UP_TOP +
                    a load-validation line corroborating RIGHT_UP_TOP.
                    Anomalies: TOP_DAMAGE (support) (HIGH).

  Inputs (read-only): GlobalTrainState, UnifiedWagonState,
  wagon_states/<feature>/<gw>.json, evidence/<gw>/<feature>/, and
  wagon_cache/<gw>/<cam_folder>/ frames.  No detector / model loads.

  Failure policy:
      Each camera's PDF is independent.  reportlab failure on one camera
      returns None for that camera only; the other three camera reports
      and the combined report are unaffected.

            status, mode = "TRAIN-STATE-NATIVE"

        On a restart, an already-completed upload/email for the current

  features/inference_lib/ -- MATURE INTELLIGENCE REFERENCE
================================================================================
================================================================================

  Every legacy heuristic was ported into this subpackage.  The
  feature processors call into these classes from a frame iterator;
  none of these modules depends on `wagon_timeline`, `process_video`,

================================================================================
  DETERMINISM GUARANTEES
================================================================================

  Given the same 4 videos, the same 8 .pt model files, the same easyocr
  model assets, and the same CLI flags, the pipeline produces
  byte-equivalent global_train_state.json AND combined_train_report.json.

      Stage 1   wagon_count uses deterministic sort keys.  YOLO is
                deterministic given fixed weights + device + half=False.

      Stage 2   single-pass per video; frame_idx is a running counter;
                cv2.imwrite is deterministic.

      Stage 3   Per-feature voting uses
                  (-count, -mean_confidence, alpha) tiebreaks.
                Kalman state is deterministic given the same
                initialization (the legacy trackers seed velocity to
                zero with high covariance).
                Hungarian assignment uses scipy.optimize.linear_sum_assignment
                which is deterministic.
                easyocr is deterministic given the same model + GPU
                flag + half flag.

      Stage 4   Fusion ingests features in fixed order
                  (door, load, damage, ocr).
                Provenance sets are sorted before serialization.

      Stage 5   PDF row order = GlobalTrainState wagon order; JSON key
                order is fixed.

      Stage 6   Uploads are idempotent (S3 PUT overwrites with identical
                content).

================================================================================
  END-TO-END DATA FLOW  (one train, one run)
================================================================================

  1. DISCOVER   train_batch_manager clusters the 4 camera videos that share a
                t=0 timestamp into one TrainBatch (batch_key = YYYYMMDD_HHMMSS).
                --auto finds them in S3 (INPUT_BUCKET + INPUT_PREFIXES);
                --local-only scans --local-inputs.
  2. WAIT/SEAL  Wait for the RIGHT_UP master; open a short support-fusion
                window so LEFT_UP / tops can improve gap recovery; then SEAL an
                immutable GlobalTrainState (Stage 1 counts + numbers + classes).
                Late cameras after the seal attach FEATURES only -- no recount.
  3. MATERIALIZE  Stage 2 slices each video into per-wagon frame caches.
  4. FEATURES   Stage 3 runs door / ocr / load / damage per wagon, writing
                per-wagon JSON + camera-scoped evidence/<GW>/<feat>/<CAM>/.
  5. FUSE       Stage 4 fuses features into one UnifiedWagonState per wagon
                (RIGHT_UP authority for identity/classification).
  6. RENDER     Stage 4b draws overlay videos (visualization only, no models).
  7. REPORT     Stage 5a -> 4 per-camera PDFs; Stage 5b -> combined PDF + JSON.
  8. DELIVER    Stage 6: ONE PDF upload + ONE email (idempotent via
                delivery/finalization.json); tree archive to S3; mark the
                batch in master_runner/processed_batches.json.  OPTIONAL:
                legacy-dashboard ingest (on by default; WAGONEYE_DASHBOARD_INGEST_ENABLED=false to skip).

  Outputs land under  batch_outputs/<batch_key>/  (downloads, global_state,
  wagon_cache, wagon_states, evidence, processed_videos, reports, delivery,
  archive) and are mirrored to s3://<output_bucket>/train_batch/<batch_key>/.

================================================================================
  WHAT EACH MODEL DOES
================================================================================

  right_up_gap.pt   / left_up_gap.pt
      YOLO detection -- inter-wagon gaps in side camera frames.
      Class: "gap".  Used by wagon_count to build the master gap
      timeline.  Stage 1 only.

  top_gap.pt
      YOLO detection -- inter-wagon gaps in top camera frames.
      Class: "gap".  Used by wagon_count for cross-camera gap-fusion
      votes.  Stage 1 only.

  side_classification.pt
      Frame classifier -- ENGINE / WAGON / BRAKE_VAN.  Drives the
      classification field on every GlobalWagon.  Stage 1 only.

  door_state.pt
      YOLO detection -- doors + state classes:
        open / open_door / closed / closed_door /
        partial_closed / closed_with_wire / damage.
      Stage 3.A feature.

  loaded.pt
      YOLO classification (results.probs.top1) -- LOADED vs EMPTY.
      Stage 3.C feature.

  damage.pt
      YOLO detection -- top damages: floor_damage / inner_wall_damage /
      no_damage (outer_wall_damage is filtered out on TOP cameras --
      it's a side concern).  Stage 3.D feature.

  wagon_id_counting.pt
      YOLO detection -- wagon-number plate regions.  Each crop is read
      by easyocr; results aggregated by exact 11-digit string with
      digit-level voting.  Stage 3.B feature.


================================================================================
  WHAT IS EXPLICITLY NOT IN THIS PACKAGE
================================================================================

  By design (per the architectural constraints):

      - No RIGHT_UP/, LEFT_UP/, RIGHT_UP_TOP/, LEFT_UP_TOP/ folders.
      - No DoorProcessor / DamageProcessor wrappers (the legacy
        process_video() entry points).
      - No camera_loader.py (legacy isolated-module loader).
      - No `cv2.VideoCapture` calls for INFERENCE anywhere downstream
        of Stage 2.  Stage 4b's overlay renderer uses cv2.VideoCapture
        for VISUALIZATION ONLY -- it consumes already-persisted state
        and evidence, and never invokes any detector model.
      - No mini-mp4 reconstruction per wagon.
      - No wagon-segmentation logic in any feature processor (wagons
        are fixed by GlobalTrainState).
      - No legacy combined_report_generator (a fresh reportlab writer
        is in reporting/combined_train_report.py).
      - No model loads inside rendering/ or reporting/.  The
        rendering/feature_overlay_renderer module imports cv2 + numpy
        only; reporting/{camera_reports,combined_train_report}.py
        import reportlab only.  Anything else would violate the
        visualization/inference separation.

  Future work (out of scope for this revision):

      - features/side_damage/ -- a side-camera damage detector using
        side_damage.pt (or equivalent).  UnifiedWagonState already
        carries side_damage / side_damage_details fields -- only the
        detector module needs to be added.
      - GPU memory pooling across feature processors.
      - Streaming wagon_cache upload to S3.
